## 프로젝트 개요

다음 조건으로 **1인용 백개먼 게임**을 구현하라.

- **TypeScript**
- **HTML5 Canvas**
- **외부 라이브러리/프레임워크 의존성 없음**
- 브라우저 기본 API만 사용
- 사람이 컴퓨터 AI와 대전하는 **싱글 플레이**
- **IndexedDB 기반 이어하기(저장/복원)** 지원
- **예외 처리와 복구 전략** 포함

결과물은 로컬에서 실행 가능한 **완결된 프론트엔드 프로젝트**여야 한다.

---

## 핵심 목표

아래를 모두 만족하는 **실제로 플레이 가능한 백개먼 게임**을 만들어라.

- 올바른 보드 배치
- 올바른 이동 규칙
- 주사위 굴리기
- 히트(hit), 바(bar), 재진입(re-entry)
- 베어오프(bearing off)
- 턴 진행 관리
- 합법 수 검증
- 간단하지만 정상 작동하는 AI 상대
- Canvas 기반 렌더링
- 마우스 입력 기반 조작
- IndexedDB 기반 자동 저장/이어하기
- 저장 데이터 손상/실패에 대한 안전한 예외 처리

## 규칙 구현 요구사항

백개먼 핵심 규칙은 **대충 처리하지 말고 정확하게 구현**하라.

### 필수 규칙 체크리스트

- [ ] 24개 포인트를 표현한다
- [ ] 각 플레이어 15개 말의 총합이 항상 유지되어야 한다
- [ ] 상대 말이 2개 이상 있는 포인트로는 이동할 수 없다
- [ ] 상대 말이 정확히 1개면 히트할 수 있다
- [ ] 히트된 말은 bar로 보낸다
- [ ] bar에 말이 있으면 그 말부터 재진입해야 한다
- [ ] 재진입도 주사위 값과 블록 규칙을 따라야 한다
- [ ] 두 주사위를 모두 사용할 수 있으면 반드시 둘 다 사용해야 한다
- [ ] 둘 중 하나만 사용할 수 있으면 하나를 사용해야 한다
- [ ] 둘 다 단독 사용 가능하지만 동시에 둘 다는 못 쓰는 상황에서는 표준 규칙에 따라 **더 큰 눈을 우선 강제**해야 한다
- [ ] doubles는 4개의 하위 이동으로 처리해야 한다
- [ ] bearing off의 정확한 규칙을 구현해야 한다
- [ ] 정확히 맞는 눈으로 bear off 가능해야 한다
- [ ] 더 높은 포인트에 말이 없을 경우, 더 낮은 포인트의 가장 높은 말을 큰 눈으로 bear off할 수 있어야 한다
- [ ] 승리 조건은 15개 모두 bear off이다

### 중요한 구현 원칙

- [ ] 단일 이동 legality만 검사하지 말고 **턴 전체의 합법적 이동 시퀀스**를 생성/검증할 수 있어야 한다
- [ ] 주사위 사용 강제 규칙을 만족하도록 설계한다
- [ ] bar / bearing off / doubles / higher-die rule을 예외 없이 반영한다

## IndexedDB 저장/이어하기 요구사항

이 요구사항은 **필수**이다. 선택사항이 아니다.

### 저장소 요구사항 체크리스트

- [ ] **브라우저 IndexedDB를 직접 사용한다**
- [ ] `localStorage`로 대체하지 않는다
- [ ] 저장 로직은 별도 모듈로 분리한다
- [ ] 예: `src/persistence/IndexedDbStore.ts`

### 이어하기 기능 체크리스트

- [ ] 게임 상태를 자동 저장한다
- [ ] 앱을 다시 열면 최근 진행 중 게임을 복원할 수 있어야 한다
- [ ] 저장된 게임이 있어도 새 게임 시작이 가능해야 한다
- [ ] 저장 데이터를 지우는 기능이 있어야 한다
- [ ] 저장된 상태로부터 게임이 **정확히 이어져야 한다**

### 반드시 저장해야 하는 상태

- [ ] 보드의 말 배치
- [ ] bar 개수
- [ ] borne-off 개수
- [ ] 현재 턴 플레이어
- [ ] 현재 주사위 상태
- [ ] 남은 주사위 사용 상태
- [ ] 현재 phase/state
- [ ] 필요 시 현재 선택 상태
- [ ] 안전한 복원을 위한 AI 진행 상태 또는 최소한의 턴 상태

### 저장 시점 체크리스트

적절한 타이밍에 자동 저장하라.

- [ ] 새 게임 시작 직후
- [ ] 주사위 굴림 직후
- [ ] 각 이동이 확정된 직후
- [ ] 턴 종료 직후
- [ ] AI 턴 완료 직후
- [ ] 게임 종료 직후

### 시작 시 복원 체크리스트

앱 시작 시:

- [ ] 저장된 진행 중 게임이 있는지 확인한다
- [ ] 있으면 안전하게 복원한다
- [ ] 저장 데이터가 손상/불일치/구버전이면 안전하게 실패 처리한다
- [ ] 실패 시 앱이 죽지 않아야 한다
- [ ] 복원 불가 시 새 게임으로 안전하게 전환할 수 있어야 한다
- [ ] 필요하면 사용자 친화적 메시지를 보여준다

### 저장 포맷 체크리스트

- [ ] 저장 데이터에 schema version을 포함한다
- [ ] timestamp를 포함한다

## 예외 처리 요구사항

이 프로젝트는 **해피패스만 구현하면 안 된다**.  
복구 가능한 오류는 최대한 복구하고, 복구 불가능하면 안전하게 초기화하라.

### 반드시 고려할 예외 체크리스트

- [ ] IndexedDB open 실패
- [ ] IndexedDB read 실패
- [ ] IndexedDB write 실패
- [ ] 손상되거나 일부만 저장된 데이터
- [ ] 잘못된 move 입력
- [ ] 내부적으로 불가능한 상태 전이
- [ ] canvas context를 얻지 못하는 경우
- [ ] null / undefined 접근 위험
- [ ] legal move가 있어야 하는데 AI가 못 찾는 경우
- [ ] 잘못된 phase에서 사용자 입력이 들어오는 경우
- [ ] 새로고침/재접속 시 턴 중간 상태 복원 문제

### 예외 처리 정책 체크리스트

- [ ] 가능한 한 앱 전체 크래시를 막는다
- [ ] 복구 가능한 오류는 사용자 친화적 메시지와 함께 복구한다
- [ ] 개발 디버깅용 콘솔 로그를 남긴다
- [ ] 복구 불가능하면 안전하게 새 게임 상태로 되돌린다
- [ ] 손상된 상태를 조용히 계속 사용하지 않는다
- [ ] 침묵하는 데이터 오염(silent corruption)을 피한다

### 코딩 원칙 체크리스트

- [ ] 브라우저 API 접근에 방어적 체크를 둔다
- [ ] 역직렬화된 데이터를 검증 후 사용한다
- [ ] fragile assumption보다 명시적 검사와 반환을 선호한다
- [ ] tricky recovery path에는 간결한 주석을 남긴다

## 수용 기준(Acceptance Criteria)

아래를 모두 만족해야 완료로 본다.

### 필수 완료 조건 체크리스트

- [ ] 외부 라이브러리 없이 브라우저에서 실행된다
- [ ] 사람이 AI와 끝까지 한 판 플레이할 수 있다
- [ ] illegal move가 방지된다
- [ ] hit / bar / re-entry가 정상 동작한다
- [ ] bearing off가 정상 동작한다
- [ ] doubles가 정상 동작한다
- [ ] 턴 진행이 안정적이다
- [ ] AI가 자신의 턴을 합법적으로 끝낼 수 있다
- [ ] 게임이 정상적인 종료 상태에 도달한다
- [ ] 게임 상태가 **IndexedDB**에 저장된다
- [ ] 앱을 닫았다 다시 열어도 **이어하기**가 가능하다
- [ ] 저장 데이터가 손상되어도 앱이 크래시하지 않는다
- [ ] 저장/로드 실패가 우아하게 처리된다
- [ ] 저장된 게임이 있어도 새 게임 시작이 가능하다
- [ ] 저장 데이터 삭제가 가능하다
- [ ] 코드베이스가 이해 가능하고 모듈화되어 있다
